# 敏捷中的各项计划
```
scrum待办事项
   一个包含所有产品需求和功能改进的列表
任务列表
   通常是通过scrum的待变事项的分解，包含具体任务的详细记录
发布计划
   确定关键里程碑的交付日期，例如首次发布时间，重大功能更新的时间节点
   明确每个发布版本中包含的主要特性和功能。这需要对产品需求进行优先级排序，将高优先级的需求纳入早期的发布版本中。
   评估每个版本的资源，包括人力，技术资源。
迭代计划
   确定短期迭代的目标和范围。选择本次迭代中要完成的任务和功能。
   分解任务并分配给团队成员。
   制定迭代的进度安排，迭代的开始结束时间，关键检查点，评审时间。
   当确定迭代计划时，可以参考发布计划。
```
# 敏捷中各角色的职责
```
产品负责人
   确保工作按照领导层的指示方向和产品愿景展开
   了解到新的需求时，将需求添加到产品待办事项列表中
   对于严重的问题时，在PBL中添加高优先级的工作项。
Scrum主管
   传播敏捷，无论直接参与哪个团队工作，都应该帮助公司采用敏捷方法。
   负责消除阻碍团队生产力的障碍。
开发团队
   如果开发团队在多次冲刺中意识到，无法实现发布的目标，可以于PO会议，确认优先级。
```
# 敏捷中的可视化工具
```
1、信息发射源
   一种可视化工具，用于像团队成员和利益相关者展示项目的关键信息。包含任务板、燃尽图、燃起图、任务板、问题列表、风险矩阵。
2、燃尽图
   剩余工作的趋势，横轴时间（一般代表一次迭代的时间），纵轴剩余工作量。
3、燃起图
   已完成工作的趋势，横轴时间，纵轴已完成工作量。
4、任务板
   包含几列，代表todo，doing，done，每个任务用卡片表示，任务描述、负责人、预计完成时间。
   开发团队也可以通过任务板来 整理工作、将剩余工作可视化。
5、看板
   包含几列，todo，doing，done，可以设置限制在制品（WIP）数量
   注重工作流程优化、跨职能团队协作、活着需要简单直观的任务管理工具
   解决由于缺乏信息导致的工作延迟或障碍打断
6、scrum 板
   项目强调scrum方法、强调迭代开发、强调自组织
7、待办事项列表
   待办事项列表的细化选择一个迭代持续时间。
   变更的也属于一项工作内容，会被PO放到PBL里，然后被确定优先级。
```

# 迭代的方法
```
1、六西格玛 DMAIC 流程 定义（Define）、测量（Measure）、分析（Analyze）、改进（Improve）、控制（Control）（偏向于质量）
2、PDCA（计划、执行、检查、行动）。（偏向于质量）
3、敏捷方法
3、精益生产方法
   1、价值流分析（Value Stream Mapping）
   2、5S 管理：整理（Seiri）、整顿（Seiton）、清扫（Seiso）、清洁（Seiketsu）、素养（Shitsuke），通过改善工作环境和工作习惯，提高工作效率和质量。
   3、看板管理（Kanban）：一种可视化的管理工具，用于控制生产和物流过程，实现准时化生产。
   4、持续改善（Kaizen）：鼓励员工不断提出小的改进建议，逐步实现流程的优化和提升。
```

# 知识点
```
通才型专家
   团队成员在具备和擅长一项专业化的技能的同时，还拥有多种技能的工作经验。
      I型人才只专注一项
      T型人才会多种技能（跨职能型的内部资源）
   如果项目中某些团队成员缺少关键主题的专业知识，可以在项目中添加跨职能型的内部资源。
迷你瀑布陷阱
   例如：收集所有需求，对所有需求设计规格，然后开发
```
# 易错题
```
   如果多个相关方抵制从预测到敏捷的变更，最佳的方案是，寻求一个愿意支持变更到敏捷的高管。
   在大型项目中，如果有多个团队处理项目，使用同一个产品待办列表，扩展整理范围，并考虑在产品待办列表中提供不同团队的视图。
   敏捷实践 使用产品列表、产品增量将工作划分为多个版本。
   原型设计：能收到关于早期需求的反馈相应的调整设计。
   确定待办事项优先级：团队成员觉得可能有延期的问题，其他团队也依赖这个用户故事，可以找PO重新确定优先级。
   管理质量：在敏捷中、较短迭代计划和执行周期，可以通过定期的回顾会议，来管理质量。
```
